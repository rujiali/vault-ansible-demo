Section 1 — Ansible Automation
SSH Key Rotation

Vault KV — SSH Key Pair Rotation at Scale

1
AAP job triggered on schedule or on-demand

Job template runs the SSH key rotation playbook across all target hosts in the inventory group.

2
Ansible generates a new ed25519 key pair per host

New private/public key pair generated in memory. Never written to disk on the AAP node.

3
New public key deployed to target host

Ansible pushes the new public key to ~/.ssh/authorized_keys and removes the old entry atomically.

4
New private key written to Vault KV

POST /v1/kv/data/ssh/<host> — only after host confirms new key works. Vault KV v2 retains full version history.

Key API Endpoints

# Store new private key
POST /v1/kv/data/ssh/<host>
  private_key: <pem>
  public_key: <pub>
  rotated_at: <ts>

# Read current key
GET /v1/kv/data/ssh/<host>

# View rotation history
GET /v1/kv/metadata/ssh/<host>

Why Vault KV for SSH Keys

  • Single source of truth for all host keys
  • Full version history — roll back if needed
  • AAP fetches key at job runtime via credential plugin
  • No static keys on AAP node or in playbooks

The client uses SSH public/private key pairs, not CA-signed certificates. Vault KV v2 stores the private key as the source of truth. The rotation pattern is: generate new pair → deploy public key to host → confirm access works → write private key to Vault. Write-after-confirm prevents locking yourself out.

Section 1 — Ansible Automation
SNMP Credential Rotation

KV-Backed Rotation via Ansible

1
Ansible reads current SNMP creds from Vault KV

GET /v1/kv/data/network/snmp/<device>

2
Ansible generates new credential

New community string or SNMPv3 auth/priv passwords generated in the playbook.

3
New credential pushed to device

Ansible configures the network device via SNMP management API or CLI.

4
New credential written back to Vault KV

POST /v1/kv/data/network/snmp/<device> — KV v2 retains version history.

Key API Endpoints

# Read current credential
GET /v1/kv/data/network/snmp/<dev>

# Write new credential (KV v2)
POST /v1/kv/data/network/snmp/<dev>
  data: { community: <new_val> }

# Read version history
GET /v1/kv/metadata/network/snmp/<dev>

Ansible Module

# community.hashi_vault
- vault_kv2_write:
  engine_mount_point: kv
  path: network/snmp/core-sw-01
  data:
    community: "{{ new_cred }}"

The write-after-deploy order is critical — write to the device first, then confirm success, then write to Vault. If the device update fails, Vault still holds the old credential. Never write to Vault before confirming the device accepted the change.

Section 1 — Ansible Automation
Break-Glass Credential Rotation

RHEL vs Windows — Break-Glass Rotation

RHEL OS Secrets Engine (Native)
1
OS Secrets Engine manages accounts natively

Vault connects to RHEL hosts via SSH using a management account.

2
Scheduled rotation via rotation_schedule

rotation_schedule="0 0 1 * *" — monthly rotation configured per account.

3
Ansible triggers ad-hoc rotation

POST /v1/os/hosts/<host>/accounts/<acct>/rotate

4
Version history retained in Vault

Previous credentials recoverable. Full audit trail of every rotation.

Windows AD Secrets Engine (Native)
1
AD Secrets Engine connects to Active Directory via LDAP

Vault binds to AD using a service account. No agent or WinRM required.

2
Break-glass account checked out from library

POST /v1/ldap/library/<set>/check-out — Vault issues a time-limited lease. Account locked to this checkout until check-in.

3
Scheduled rotation via ttl

Vault rotates the AD account password on a configured schedule. No manual intervention required.

4
Check-in triggers immediate rotation

POST /v1/ldap/library/<set>/check-in — password rotated immediately after session ends. Account available for next checkout.

Both RHEL and Windows break-glass are now Vault-native. RHEL uses the OS Secrets Engine (connects via SSH). Windows uses the AD Secrets Engine (connects via LDAP). Neither requires AAP for rotation — Vault owns the full lifecycle for both. The library checkout pattern is ideal for break-glass: account is reserved for the duration of the session and auto-rotated on check-in.

All Ansible → Vault calls use AppRole authentication. The Role ID is stored in the playbook; the Secret ID is injected at runtime from a secure store — no static tokens in source control.

AppRole is the right auth method for machine-to-machine (Ansible → Vault). Response-wrapping the Secret ID at delivery time gives you a one-time-use token that limits exposure if intercepted.

Break-Glass Access Workflow

End-to-end privileged access — request, authorise, audit, rotate

Transition to section 2. This covers the human workflow — how a user requests access to a break-glass account, how it gets approved, and what happens afterwards.

Section 2 — Break-Glass
User Workflow

End-to-End Break-Glass Access

1
User requests access to privileged account

Calls GET /v1/kv/data/breakglass/<host> or GET /v1/os/hosts/<h>/accounts/<a>/creds. Vault does not return the credential — returns a wrapping token.

2
Control Group holds the request

Wrapping token sent out-of-band to approver(s). TTL clock starts (e.g. 1 hour).

3
Approver(s) authorise via Vault

POST /v1/sys/control-group/authorize with the accessor. If min_approvals=2, two separate approvers required. Approver identity recorded in audit log.

4
User unwraps and accesses credential

vault unwrap <wrapping_token> — only succeeds after all required approvals. Credential is one-time readable.

5
Credential rotated immediately after use

Session ends → rotation triggered. RHEL: Vault OS Engine rotates via SSH. Windows: Vault AD Engine rotates via LDAP on check-in. Old password invalidated. See next slide.

What is Audited

  • Who requested access — identity + timestamp
  • Who approved — each approver's identity
  • When the token was unwrapped
  • When the credential was rotated post-use

Multi-Authorisation Config

control_group = {
  ttl = "1h"
  factor "dual-approval" {
    identity {
      group_names = ["security-team"]
      approvals = 2
    }
  }
}

Applies to Both RHEL & Windows

Control Group is a Vault policy feature — applies regardless of OS. The credential path changes, the approval flow is identical.

The wrapping token is the key mechanism — the requestor never sees the credential until approval is granted. The TTL means if the approver doesn't act within the window, the request expires and must be re-raised. This is a hard control, not a soft workflow.

Section 2 — Break-Glass
Post-Use Rotation

How the Credential is Rotated After Use

RHEL

OS Secrets Engine — Vault-Native

1
Session ends

Requestor notifies security team or AAP job is triggered on session close.

2
Vault rotates directly on RHEL host

vault write -f os/hosts/<h>/accounts/breakglass-rhel/rotate — Vault connects to the host via SSH and changes the password. No human sets the new value.

3
Old password invalidated immediately

Any attempt to use the prior credential fails at the OS level. New credential is stored only in Vault.

WIN

AD Secrets Engine — Vault-Native

1
Session ends

Requestor checks the AD account back in via Vault.

2
Vault rotates the AD account password directly

vault write -f ldap/library/<set>/check-in — Vault connects to AD via LDAP and rotates the password. No AAP, no WinRM.

3
Old password invalidated immediately

AD enforces the new password. Account available for next break-glass checkout.

Vault-native — same model as RHEL. No AAP dependency for rotation.

Both RHEL and Windows break-glass are now Vault-native. RHEL: OS Secrets Engine rotates via SSH. Windows: AD Secrets Engine rotates via LDAP. Neither requires AAP. The library checkout model is the right pattern for break-glass — account is reserved for the session and auto-rotated on check-in, closing the audit trail cleanly.

Section 2 — Break-Glass
Offline Access

Break-Glass When Vault is Unavailable

Risk

Vault Unavailable = No Dynamic Access

If Vault is sealed, unreachable, or in DR failover, the standard Control Group workflow cannot complete. This is a deliberate security boundary.

Mitigation

Emergency Static Credentials

A small set of static break-glass credentials are stored outside Vault — in a physically secured location (e.g. sealed envelope, privileged access workstation, or offline secrets manager) — for genuine Vault-down scenarios.

Recovery Procedure

After Vault Restores

Any static credential used during the outage must be rotated immediately once Vault is available. Usage of offline credentials must be logged manually and reviewed in the post-incident process.

Recommended Offline Credential Pattern

1
Generate static emergency creds

Created during provisioning, stored in sealed physical custody.

2
Dual-custody physical access

Two people required to access the physical store — mirrors the Control Group principle offline.

3
Vault-down declared by incident process

Offline access only authorised after formal Vault-unavailable declaration.

4
Rotate and audit on Vault restoration

Offline creds rotated in Vault immediately. Access logged in incident record.

This is a governance and process question as much as a technical one. The Design Authority should confirm: who holds physical custody of offline creds, what declares a "Vault unavailable" event, and what the post-incident rotation SLA is.

DR Replication

Full data replication to DR cluster. DR cluster is passive — cannot serve requests until promoted. Intended for full cluster failure.

Performance Replication

PR clusters can serve local read requests. Writes are forwarded to primary. Intended for latency and locality, not full DR.

DR Activation Steps
1
Declare primary failure

Confirm primary is unrecoverable. Do not promote DR prematurely — split-brain risk.

2
Generate DR operation token on primary

vault operator generate-root -dr-token — or use existing DR secondary token if primary is unreachable.

3
Promote DR cluster to primary

vault operator raft promote-dr on the DR cluster using the DR operation token. DR cluster becomes the new primary.

4
Update DNS / load balancer

Point the Vault service endpoint to the new primary. This is not automatic — requires manual DNS or LB change.

5
Re-point PR clusters

PR clusters must be updated to replicate from the new primary. See next slide.

Key point for Design Authority: the DR promotion is a manual, deliberate operation — Vault does not auto-promote. This is by design to prevent split-brain. The DNS/LB cutover is also manual and should be in the runbook.

Section 3 — Disaster Recovery
Cluster Re-pointing

Redirecting PR Clusters to the New Primary

Important

Not Automatic

Performance Replication clusters do not automatically re-point to the new primary after a DR failover. This is a manual step.

1
PR cluster detects primary is unreachable

PR cluster enters degraded mode — local reads continue, writes fail (no primary to forward to).

2
Demote and re-enable replication on PR cluster

vault write -f sys/replication/performance/secondary/disable

3
Generate new secondary activation token on new primary

vault write sys/replication/performance/primary/secondary-token id=<cluster>

4
PR cluster activates against new primary

vault write sys/replication/performance/secondary/enable token=<token> primary_api_addr=<new-primary>

During Failover Window

  • PR cluster reads: continue (cached locally)
  • PR cluster writes: fail (no primary to forward to)
  • New secret creation: unavailable
  • Existing secret reads: available

Automation Recommendation

The re-pointing steps can be scripted and triggered as part of the DR runbook. Vault does not orchestrate this automatically — the runbook should include the re-point commands as a mandatory post-promotion step.

Recovery Time Considerations

  • DNS/LB cutover: minutes (manual)
  • PR re-point: minutes per cluster (scripted)
  • Full replication sync: depends on data volume

The "not automatic" point is the most important takeaway for the Design Authority. They need to include PR re-pointing in the DR runbook and test it during DR drills. Clients often assume this is automatic.

Local Clusters & Autonomous Operation

Secret management when local clusters operate disconnected from primary

Transition to section 4. This is about how Performance Replication clusters behave when they lose connectivity to the primary.

Section 4 — Local Clusters
Autonomous Operation

How Local Clusters Behave When Disconnected

What Continues Working

  • Reading existing secrets (locally cached)
  • Token authentication (local tokens)
  • Local policy enforcement
  • Lease renewal for existing credentials

What Stops Working

  • Writing new secrets (forwarded to primary — fails)
  • New dynamic credential generation
  • Policy changes (replicated from primary)
  • Auth method changes

Write Forwarding

By default, writes on a PR cluster are forwarded to the primary. If the primary is unreachable, writes fail. Local writes are not possible in standard PR mode.

Secret Management Model

Secrets are centrally managed in the Primary and replicated to PR clusters. Updates cannot originate from a PR cluster — they must be written to primary and will propagate.

Primary
Write here
PR Cluster A
Read only
Primary
Write here
PR Cluster B
Read only

Conflict Detection

Because writes can only originate from the primary, write conflicts do not occur in standard PR mode. There is no merge or conflict resolution needed on reconnect — the PR cluster simply resumes receiving replication from primary.

Key message: PR clusters are read-optimised secondaries, not independent nodes. The "no write conflicts" point is important — the architecture prevents conflicts by design. This is simpler than it sounds but means the PR cluster cannot be used as a write endpoint during a primary outage.

Section 4 — Local Clusters
Propagation & Resync

How Updates Propagate & What Happens on Reconnect

Central Primary, Push to Local

All secret updates originate at the Primary. The PR mechanism continuously streams changes to connected PR clusters in near-real-time. There is no pull — the primary pushes.

1
Secret updated on Primary

Write lands on primary. WAL (Write Ahead Log) entry created.

2
Primary streams WAL to connected PR clusters

Replication is continuous. Connected clusters receive changes within seconds.

3
PR cluster disconnects

Primary queues WAL entries. PR cluster serves stale-but-last-known data locally.

4
PR cluster reconnects

Primary replays queued WAL entries to the PR cluster. No manual intervention. Cluster catches up automatically.

What "Conflicts" Means Here

In Vault PR, there are no write conflicts because PR clusters cannot write independently. The only data divergence possible is:

  • PR cluster serving slightly stale reads during disconnect
  • WAL queue growing on primary during extended outage

On reconnect, the PR cluster fully resyncs — no manual merge required.

Staleness Consideration

During disconnect, PR cluster reads may be stale. For break-glass credentials rotated on primary during the outage, the PR cluster will serve the old credential until reconnect. Design Authority should consider whether break-glass access should always be directed to the primary.

The staleness point about break-glass credentials is a real architectural concern — if a credential is rotated on primary and the PR cluster is disconnected, local reads return the old credential. For break-glass specifically, consider routing access requests directly to primary or the DR cluster rather than local PR clusters.

Section 5 — Best Practices
Operational Procedures

Unseal, Backup & Restore

Unseal on Reboot
1
Auto-Unseal (recommended)

Vault configured with HSM / cloud KMS Transit Seal. On restart, Vault automatically unseals by calling the seal provider. No human intervention required.

2
Manual Unseal (fallback)

If auto-unseal provider is unavailable, manual unseal using Shamir key shares. Requires quorum of key holders (e.g. 3 of 5).

3
Unseal key governance

Shamir shares distributed to separate custodians. No single person holds quorum. Shares stored offline, not in Vault itself.

4
Verify unseal via health check

GET /v1/sys/health — returns sealed status. Monitor for unexpected seal events.

Backup & Restore
1
Integrated Raft snapshot (recommended)

vault operator raft snapshot save snapshot.gz — consistent snapshot of all Vault data. Schedule regularly via cron or CI.

2
Store snapshot securely

Snapshot contains encrypted Vault data. Store in secured object storage. Encrypt at rest. Retain multiple versions.

3
Restore from snapshot

vault operator raft snapshot restore snapshot.gz — restores full state. Vault must be initialised but can be empty.

4
Test restores regularly

Backup is only valid if restore is tested. Include restore drill in DR testing schedule.

Two key messages for Design Authority: (1) Auto-unseal is not optional at scale — manual unseal with Shamir shares is operationally fragile at 3am. (2) Snapshots are encrypted but must still be secured — they contain all Vault data. Test restores are mandatory, not optional.

Summary

Key Architecture Decisions

Ansible + Vault

Vault is source of truth. Ansible is executor. AppRole auth — no static tokens in playbooks.

Break-Glass

Control Groups enforce dual approval. RHEL via OS engine. Windows via LDAP engine (Vault-native). Offline credentials in physical custody.

DR

Promotion is manual — prevents split-brain. PR cluster re-pointing is not automatic. Include both in DR runbook and test regularly.

Local Clusters

Writes always originate at primary. No conflicts by design. Consider routing break-glass reads to primary, not PR clusters.
